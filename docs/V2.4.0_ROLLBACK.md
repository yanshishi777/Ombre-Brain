# OmbreBrain v2.4.0 Rollback Guide

This guide covers local rollback only. No cloud server operation is required.

## What v2.4.0 Adds

v2.4.0 adds code under `src/ombrebrain/`, docs, tests, dashboard debug views, and local artifacts. Runtime diagnostic state is stored in the local side-channel directory:

`.ombrebrain-v3`

That side-channel is separate from user-facing config, buckets, and vector database state.

## Safe Rollback Order

1. Stop local OB processes.
2. Preserve user data first: config, buckets, vector database files, and deployment overrides.
3. Save any useful artifacts, such as generated diff files, health reports, and zip archives.
4. Remove local diagnostic state only if needed:

```powershell
Remove-Item -LiteralPath ".ombrebrain-v3" -Recurse -Force
```

5. Revert local code with git only after reviewing the exact diff. Do not use destructive commands against unknown paths.
6. Re-run:

```powershell
py -3.10 -m pytest -q
```

## Git Safety

- No commit is required for rollback.
- No push is required for rollback.
- No release or tag is required for rollback.
- Use git status and git diff to inspect changes before reverting.
- Keep artifacts if they help compare v2.3.22 and v2.4.0 locally.

## Protected State

Do not delete or rewrite these as a rollback shortcut:

- config
- buckets
- vector database files
- OAuth secrets
- deployment user overrides

If rollback needs to restore old code behavior, revert code and remove `.ombrebrain-v3` first. User memory data should remain untouched.

## Remote Safety

- No cloud server change.
- No SSH action.
- No GitHub push.
- No GitHub release.
- No production redeploy.

Rollback is a local working-copy decision until the maintainer explicitly chooses otherwise.
