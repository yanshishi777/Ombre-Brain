# Ombre Brain v2.4.0 Release Notes Draft

## v2.4.0 Foundation

v2.4.0 keeps the original Ombre Brain purpose intact: durable emotional memory
for Claude and other MCP clients. The new architecture adds a deeper internal substrate
without changing the existing v2.3.22 runtime entry points.

Included in this local v2.4.0 work:

- collaboration graph for parent/child memory provenance, source chains, actor views,
  and merge decisions
- Raft-style local cluster simulator with majority election, leader commit, follower
  log replication, and minority partition rejection
- snapshot and catch-up primitives with checksum verification and strict event id checks
- capability catalog for MCP aggregation, tool search, breath tools, dashboard, OAuth
  control, hot update planning, and local deployment
- manifest-driven hot update policy that rejects `.env`, `config.yaml`, bucket data,
  vector database paths, OAuth secrets, unsafe paths, and deployment user overrides
- v2 bucket migration report for dynamic, permanent, archive, feel, plans, and letters
- v2.4.0 noncommercial source notice for public inspection and noncommercial self-hosting

## Runtime Hardening

After the foundation work, v2.4.0 now routes legacy behavior through deeper internal
diagnostics while preserving the public OB contract:

- command and projection layers record typed plans, consistency reports, and Projection observers
- Policy VM and capability contracts classify protected surfaces without changing user calls
- DecisionLedger writes JSON-safe DecisionRecord metadata into the local v2.4.0 side channel
- Decision Debug Service lists recent records and reports malformed decision metadata
- DecisionReplayService rebuilds decision summaries without re-running legacy actions
- 2.4.0 Debug Dashboard exposes local read-only debug views for decision and replay inspection

## Final Hardening: F -> G -> H -> I -> J

The final local pre-release pass adds:

- F: `ArchitectureAuditor`, `ComponentGraph`, and `ArchitectureReport` to verify critical
  component boundaries, ownership, side-effect modes, and dependency health
- G: `V3ResilienceScanner` and `ResilienceReport` to replay the local WAL, inspect
  decision debug output, and catch metadata shape drift without mutating state
- H: `V3MaintenanceReportBuilder` and `tools/v3_health_report.py` to package architecture,
  resilience, runtime, and recent decision data into one offline report
- I: architecture and boundary docs that pin read-only, audit-only, write-side-channel,
  and protected surface rules
- J: pre-release acceptance, rollback, and release-note coverage for local verification

## Deep Kernel Sequence: F -> B -> C -> D -> E -> A

This local v2.4.0 line also includes the approved deep-kernel sequence while preserving the internal architecture package names:

- F: `FormalAcceptanceHarness` for machine-checkable legacy compatibility
- B: `EventSourcedMemoryKernel` for command-event-projection side-channel metadata
- C: `QueryPlanner` and `RetrievalEngine` for explainable breath/search planning
- D: `CapabilityMicrokernel` for centralized capability permission and side-effect decisions
- E: `PluginRuntime` and `PluginSandbox` for constrained extensions
- A: `DistributedMemoryFabricCluster` for membership, leader lease, quorum commit, follower read, and catch-up primitives

## Compatibility

The v2.4.0 modules are additive under `src/ombrebrain/`. MCP tool names, tool parameters,
return text, bucket markdown, existing Dashboard routes, config meaning, and deployment
override behavior are intended to remain unchanged.

## Pre-Release Acceptance

This draft is a pre-release acceptance artifact only. Before publishing manually, run:

```powershell
py -3.10 -m pytest -q
```

Also review `docs/V2.4.0_ACCEPTANCE_CHECKLIST.md`, `docs/V2.4.0_ROLLBACK.md`,
`docs/V2.4.0_ARCHITECTURE.md`, and `docs/V2.4.0_BOUNDARY_MAP.md`.

No commit, push, release, tag, package publication, SSH action, or cloud server change
is implied by this draft.

## Future Work

The full multi-node production transport remains future work. This draft includes
local consensus, replication, snapshots, policy primitives, diagnostics, and local
acceptance tooling; production networking, node discovery, durable membership changes,
cross-host secure transport, and live rolling updates still need a dedicated
implementation phase.
