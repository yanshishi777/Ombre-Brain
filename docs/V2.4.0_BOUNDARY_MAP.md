# OmbreBrain v2.4.0 Boundary Map

This map fixes the boundary between original OB behavior and the v2.4.0 architecture. The goal is deeper internals with the same external function.

## Protected Surfaces

The protected surfaces are:

| Surface | Boundary rule |
| --- | --- |
| config | v2.4.0 architecture may inspect settings and classify risk; it must not rewrite user config from audit/debug paths. |
| buckets | legacy bucket code remains the owner; v2.4.0 architecture may observe bucket metadata and consistency only. |
| vector database | embedding and vector storage behavior remains legacy-owned; v2.4.0 architecture may report shape and policy context. |
| OAuth secrets | v2.4.0 debug and maintenance tools are read-only and must not expose secret values. |
| deployment overrides | deploy files and user overrides remain user-controlled. |
| dashboard routes | existing Dashboard routes continue to work; v2.4.0 adds debug routes without replacing old endpoints. |
| MCP tools | old MCP names, input parameters, and return text are kept stable. |

## Boundary Modes

| Mode | Allowed | Forbidden |
| --- | --- | --- |
| read-only | inspect files, replay architecture WAL, list decision summaries, render debug state | append events, mutate config, mutate buckets, call external services |
| audit-only | build `CommandPlan`, policy verdicts, projection observations, `ArchitectureReport` | claim ownership of legacy files or change public return values |
| write-side-channel | append architecture trace metadata to the local `MemoryFabric` WAL | rewrite bucket markdown, vector database content, OAuth secrets, or deploy settings |
| write-legacy-state | legacy OB modules already owning old behavior | new v2.4.0 diagnostics taking over old state |

## Legacy Compatibility Contract

- MCP tool names unchanged.
- MCP arguments and return text unchanged.
- bucket markdown unchanged.
- Dashboard existing routes unchanged.
- Config and environment variable names keep their old meaning.
- Deployment templates keep user override behavior.
- No cloud server changes.
- No commit, push, release, or publish is implied by v2.4.0 hardening work.

## Component Boundary Table

| Component | Boundary |
| --- | --- |
| `ExecutionEnvelope` | Captures caller, permissions, payload, and protected paths without executing legacy work. |
| `CommandPlan` | Gives old operations a typed plan used by policy, projection, and decision layers. |
| `Projection observers` | Observe bucket, trace, policy, and tool surfaces; they report consistency without mutating source data. |
| `Policy VM` | Converts module profiles and protected surfaces into deterministic policy verdicts. |
| `DecisionRecord` | Freezes policy, projection, replay, and capability metadata into a JSON-safe record. |
| `DecisionDebugService` | Reads recent records and reports malformed metadata. |
| `DecisionReplayService` | Replays decision summaries from recorded metadata only. |
| `ComponentGraph` | Lists architecture components, ownership, dependencies, and side effect modes. |
| `ArchitectureReport` | Proves component boundaries are still coherent. |
| `ResilienceReport` | Proves local WAL, decision records, and metadata shapes are readable. |
| `V3MaintenanceReportBuilder` | Packages read-only health data for maintainers. |
| `FormalAcceptanceHarness` | Verifies legacy tool, route, bucket markdown, and protected surface compatibility. |
| `EventSourcedMemoryKernel` | Emits command-event-projection side-channel metadata without rewriting buckets. |
| `QueryPlanner` / `RetrievalEngine` | Plans and explains retrieval without changing MCP return text. |
| `CapabilityMicrokernel` | Centralizes permission checks and side-effect classification for capabilities. |
| `PluginRuntime` | Runs only declared plugin handlers allowed by sandbox policy. |
| `DistributedMemoryFabricCluster` | Coordinates local quorum commit, leader lease, follower reads, and catch-up for architecture events. |

## Safe Local Maintenance

The maintainer path is local:

```powershell
py -3.10 tools/v3_health_report.py --buckets-dir buckets --output reports/v3-health.json
```

This path is read-only for legacy state. It can create the requested report file, but it does not change config, buckets, vector database files, OAuth settings, deployment files, GitHub state, or cloud servers.

## Hot Update Boundary

The v2.4.0 policy and architecture layers can classify protected update targets, but hot update execution must continue to exclude user data and protected surfaces. In particular, config, buckets, vector database files, OAuth secrets, and deployment overrides are not overwritten by diagnostic or maintenance tools.

## Deep Kernel Compatibility Boundary

The F -> B -> C -> D -> E -> A sequence is allowed to deepen internals but not change public behavior. Event sourcing, query planning, microkernel dispatch, plugins, and distributed fabric all remain behind the compatibility contract:

- old MCP calls still use their current names and argument shape
- old bucket markdown remains readable
- old Dashboard routes remain available
- plugin code cannot write protected legacy surfaces directly
- distributed fabric is an internal side-channel until an explicit migration plan says otherwise
